# Churn Prediction Project

Проект по прогнозированию оттока клиентов. Включает полный цикл:

- EDA,
- обработку сырого датасет,
- обучение модели (CatBoost),
- подбор порога классификации,
- инференс
- анализ ошибок и выводы.

## Установка

Советую использовать uv для создания виртуального окружения и установки зависимостей.

## Запуск

1. Подготовка данных

   ```bash
   python src/make_dataset.py
   ```

   Создаёт `data/processed/processed_data.csv` — агрегированный датасет на уровне клиента.

2. Обучение модели

   ```bash
   python src/train_model.py
   ```

   Обучает CatBoost с подбором гиперпараметров, вычисляет оптимальный порог (максимизирующий F1), сохраняет модель и порог в `models/`.

3. Инференс

   ```bash
   python src/predict.py
   ```

   Вернёт JSON с вероятностью оттока и бинарным предсказанием.

   Запустите `notebooks/error_analysis.ipynb` для визуализации метрик, важности признаков и анализа ошибок.

## Результаты

- **ROC-AUC:** ~0.70
- **Recall при оптимальном пороге:** ~99%
- **Precision:** ~14%

Модель находит почти всех уходящих клиентов, но даёт много ложных  срабатываний. Это может быть приемлемо, если бизнес готов предложить  удержание широкому кругу клиентов. При необходимости порог можно  пересчитать под другие бизнес-метрики (например, под заданный  минимальный precision).

## Примечания

- Данные после анализа оказались слабо информативными (признаки статичны,  корреляции с оттоком близки к нулю), что объясняет ограниченное качество модели.
- В текущей реализации я использовал **агрегацию на уровне клиента**, так как `churn` оказался статической меткой (клиент либо ушёл за весь период, либо  нет). Однако в реальном бизнесе данные постоянно обновляются, и отток  обычно прогнозируется на **следующий месяц** на основе истории поведения. Поэтому для реального проекта архитектуру обработки данных я бы изменил:
  1. **Отказаться от агрегации клиента** — сохранить панельную структуру (одна строка — один месяц для клиента).
  2. **Создать лаговые признаки** — для каждого динамического признака добавить значения за предыдущие 1, 2, 3 месяца (например, `unpaidbalance_lag1`, `unpaidbalance_lag2`).
  3. **Скользящие статистики** — средние, стандартные отклонения за последние 3–6 месяцев.
  4. **Признаки изменений** — разница между текущим и предыдущим месяцем, процентное изменение.
  5. **Целевая переменная** — `churn_next_month`: ушёл ли клиент в следующем месяце (сдвиг метки во времени). Это позволяет строить прогноз на упреждение.
  6. **Валидация по времени** — обучать модель на данных до определённой даты, тестировать на более поздних периодах.
